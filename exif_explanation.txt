EXIF PARSING IN MORPHGUARD SYSTEM
==================================

EXIF Parsing refers to the extraction and analysis of Exchangeable Image File Format (EXIF) metadata embedded inside digital image files (like JPEGs and TIFFs). Devices like cameras and smartphones automatically write this data when an image is created. It includes settings (shutter speed, aperture), camera hardware (Make, Model), timestamps, software used, and sometimes GPS location.

In digital forensics, when an image is manipulated in software (like Photoshop or MS Paint), the software often leaves traces in the EXIF data—either by overriding the `Software` tag with its own name, messing up the original dimensions, or completely stripping the EXIF data away.

Here is a breakdown of the specific criteria, formulas, and academic context used in your project's `exif_analysis.py` file.


CRITERIA AND SCORING FORMULA
============================
Your implementation extracts the raw EXIF tags using the Python Imaging Library (PIL) (`img._getexif()`) and maps them to human-readable tag names. It then evaluates the data against 5 specific forensic criteria and accumulates a `score` representing the image's suspicion level:

1. Missing EXIF Pipeline (Score: +15 points)
   - Code Formula: `if exif_data is None:`
   - Logic: Original photos straight from a camera always contain robust EXIF blocks. If it is entirely missing, the image was likely exported by a generic tool, screenshot software, or a basic bitmap editor (like MS Paint or a web browser right-click save), which strip this data.

2. Editing Software Signatures (Score: +30 points)
   - Code Formula: `if editor in combined:` (where `combined = software + " " + processing_software`)
   - Logic: The script checks the EXIF `Software` and `ProcessingSoftware` tags against an internal dictionary of 20 known editing suites (e.g., "photoshop", "gimp", "canva", "lightroom"). If a match is found, it inherently proves the image was saved by photo manipulation software rather than a camera sensor.

3. Spatial/Dimension Mismatch (Score: +20 points)
   - Code Formula: `if (exif_w != actual_w or exif_h != actual_h):`
   - Logic: When an image is cropped or resized, cheap editing software modifies the actual image pixel grid but often forgets to mathematically update the `ExifImageWidth` or `ExifImageHeight` metadata tags. If the physical pixel dimensions don't match the metadata dimensions, the image has strictly been tampered with.

4. Missing Camera Hardware Info (Score: +5 points)
   - Code Formula: `if not metadata.get("Make") and not metadata.get("Model"):`
   - Logic: If EXIF data exists but strictly lacks the `Make` (e.g., Apple, Canon) and `Model` (e.g., iPhone 13) tags, it suggests the EXIF header was artificially generated or sanitized by an exporter program rather than originating from real hardware.

5. File Format Mismatch (Score: +15 points)
   - Code Formula: `if ext in [".jpg", ".jpeg"] and img_format not in ["jpeg", "jpg"]:`
   - Logic: Malicious morphing tools sometimes forcibly change a file extension (e.g., renaming `fake.png` to `real.jpg`) to bypass upload filters. The script checks the actual byte-header signature (`img.format`) against the file extension to ensure integrity.

   - Relevance to your project: While primary focused on JPEG compression, it relies heavily on metadata discrepancies (like your Spatial/Dimension Mismatch criteria) to prove that a sub-section of an image was copied from a different source.

3. "Image Forgery Detection based on Image Quality Metrics and Meta-Data Analysis" 
   - Relevance to your project: Many modern survey papers discuss exactly what you have implemented—using a weighted scoring system across Software, Make/Model, and ImageSize mismatches to create a generalized suspicion score for digital certificates and photographs.
